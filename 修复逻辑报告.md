# Winetricks 修复 Proton / Wine 游戏的逻辑分析报告

> 分析对象：`fixing_tools/winetricks`（主体是 `src/winetricks`，一个约 2 万行的 POSIX shell 脚本）
> 一句话定位：Winetricks 是面向 Wine prefix 的 **Win32 DLL / 应用"包管理器"**，通过下载、校验、安装 Windows 运行库/DLL/字体，并设置 DLL override 与注册表，来修复游戏在 Wine/Proton 下的兼容性问题。

---

## 1. 整体定位

脚本自我描述为：*"Winetricks is a package manager for Win32 dlls and applications on POSIX."*

核心职责：

| 能力 | 说明 |
|------|------|
| 自动下载 | 从 Microsoft、GitHub 等可信源拉取运行库、DirectX、.NET、字体等 |
| 安装到 prefix | 解压 / 运行安装程序，把 DLL、字体、程序写入 `WINEPREFIX` 的虚拟 C: 盘 |
| Wine 配置 | 通过注册表设置 DLL override、Windows 版本、图形/字体 tweak |
| Bug 规避 | `w_package_broken`、`w_workaround_wine_bug` 等按 Wine 版本给出警告或绕过 |
| 交互方式 | CLI（`winetricks vcrun2019 d3dx9`）或 GUI（zenity / kdialog） |

**与 Proton 的关系**：Winetricks 本身**不感知 Proton**，它只依赖环境变量 `WINE`、`WINEPREFIX`、`WINESERVER`。在 Proton 生态里，由 **Protontricks**（或 umu-protonfixes 内部）负责定位 Steam 游戏 prefix、注入 Proton 版的 Wine 与 Steam Runtime，然后调用 winetricks 真正完成组件安装。

## 2. Verb（动词 / 组件）机制

### 2.1 组织结构
每个 verb 由**一对相邻定义**组成：

```sh
w_metadata <verb_name> <category> \
    title="..." publisher="..." year="..." \
    media="download" \
    file1="..." \
    installed_file1="..." \
    conflicts="..."   # 可选

load_<verb_name>() { ... }   # 实际安装逻辑
```

运行时调度链：
```
execute_command(verb)
  → w_call(verb)          # 包装错误处理
    → w_do_call(verb)     # 子 shell 隔离环境，防止 verb 互相污染
      → 加载 metadata(.vars)
      → winetricks_set_wineprefix()
      → 检查 conflicts / 是否已安装
      → load_<verb>()     # 执行安装
      → winetricks_stats_log_command()  # 记录到 winetricks.log
```

### 2.2 Metadata 存储
- `w_metadata()` 在运行时把元数据写入临时目录 `WINETRICKS_WORKDIR/metadata/<category>/<verb>.vars`。
- 静态列表见 `files/verbs/*.txt`（如 `dlls.txt`、`settings.txt`），供文档/测试用。
- 支持外部 `.verb` 文件扩展。

常用 metadata 字段：`title`/`publisher`/`year`（展示）、`media`（`download`/`manual_download`）、`file1...`（下载文件名）、`installed_file1`/`installed_exe1`（安装验证路径）、`conflicts`（互斥）、`category`。

### 2.3 分类（`winetricks_init()` 硬编码）
```sh
WINETRICKS_CATEGORIES="apps benchmarks dlls fonts settings mkprefix"
```

| 分类 | 典型内容 | 说明 |
|------|----------|------|
| **dlls** | vcrun2019, d3dx9, dxvk, dotnet48 | 运行库、图形层、.NET（修复游戏最常用） |
| **fonts** | corefonts, droid | 字体安装 + 注册表注册 |
| **settings** | win7, csmt=on, grabfullscreen=y | 只改 Wine 设置，不下载大文件 |
| **apps** | steam, firefox, 7zip | 完整 Windows 应用 |
| **benchmarks** | 3dmark06, unigine_heaven | 基准测试程序 |
| **mkprefix** | 创建/选择 prefix | prefix 管理 |

> 没有独立的 `games` 分类，游戏相关 DLL（如 `cnc_ddraw`）归入 dlls，完整游戏安装归入 apps。

### 2.4 特殊 verb 形式
- `native d3dx9_43` / `builtin foo` / `disabled bar`：直接调用 `w_override_dlls`，无需 `load_*`。
- `settings` 类常用 `verb=值`，如 `csmt=on`、`win7`。
- `prefix=名称`、`arch=win32`：prefix 与架构管理。

## 3. 下载机制

调用链：
```
w_download url [sha256] [filename]
  → w_download_to(W_PACKAGE, ...)
    → winetricks_download_setup()   # 选择 aria2c/wget/curl/fetch
    → 缓存到 W_CACHE/<packagename>/<filename>
```

- **缓存目录**：默认 `~/.cache/winetricks/`（`XDG_CACHE_HOME/winetricks`），可用 `W_CACHE` 覆盖。
- **SHA256 校验**：`w_verify_shasum()` → `w_verify_sha256sum()`。下载前若缓存 checksum 不匹配则备份重下；下载后再次校验，失败可用 `--force` 跳过。
- 其他方式：`w_download_manual()`（手动放入缓存）、`w_download_torrent()`、`w_linkcheck_ignore=1`（跳过 linkcheck，如 dxvk 动态最新版 URL）。

## 4. DLL 安装与 Override（修复核心）

### 4.1 物理安装
在 `load_*` 内常用：
- `w_try_cabextract`：从 EXE/CAB 提取 DLL 到 `W_SYSTEM32_DLLS` / `W_SYSTEM64_DLLS`。
- `w_try_ms_installer` / `w_try "${WINE}" setup.exe`：运行 Windows 安装程序。
- `w_try mv`：直接复制（如 DXVK 的 x32/x64 DLL）。
- `w_try_regsvr32`：注册 COM 组件。

路径变量（由 `winetricks_set_wineprefix()` 按架构设置）：

| 架构 | 32 位 DLL 目录 | 64 位 DLL 目录 |
|------|----------------|----------------|
| win64 | `syswow64` | `system32` |
| win32 | `system32` | （不存在） |

### 4.2 DLL Override（Wine 加载策略）
`w_override_dlls mode dll1 dll2 ...`：
1. 生成 `override-dll.reg`，写入 `HKEY_CURRENT_USER\Software\Wine\DllOverrides`。
2. 每个 DLL 写入 `"*module"="mode"`（`*` 表示连绝对路径加载也 override）。
3. `w_try_regedit()` 导入注册表。

Override 模式：

| 模式 | 含义 |
|------|------|
| `native` | 优先加载 prefix 中的原生 Windows DLL |
| `builtin` | 优先用 Wine 内置实现 |
| `native,builtin` | 先 native 失败再 builtin（vcrun 常用） |
| `disabled` | 禁用该 DLL |
| `default` | 删除 override 项 |

相关：`w_common_override_dll()`、`w_override_all_dlls`/`w_override_no_dlls`、`w_override_app_dlls`（针对特定 exe）。

## 5. 与 Wine 的交互

环境变量：`WINE`、`WINE64`、`WINE_ARCH`、`WINEPREFIX`、`WINESERVER`、`WINEDLLOVERRIDES`（临时 override，如装 dotnet 时 `fusion=b`）、`WINEARCH`（建 prefix 用）。

关键操作函数：

| 函数 | 行为 |
|------|------|
| `winetricks_early_wine()` | 静默执行 wine（`WINEDEBUG=-all`），用于 winepath/winecfg |
| `w_try_regedit()` | win64 下用 32/64 位 regedit 双写注册表 |
| `w_set_winver()` | `winecfg -v win7` 等，设置报告给应用的 Windows 版本 |
| `w_wineserver -w` | 等待 wineserver，避免竞态 |
| `w_register_font()` | 写 Fonts 注册表项 |
| `wineboot` | 创建/初始化 prefix |

**安装记录**：成功执行 verb 后 `winetricks_stats_log_command()` 追加到 `$WINEPREFIX/winetricks.log`。`winetricks_is_installed()` 同时检查 verb 是否在 log 中、且 `installed_file1`/`installed_exe1` 真实存在（非 "Wine placeholder DLL"）。

## 6. 32 / 64 位 Prefix 处理

- 创建/检测：`winetricks_set_winearch(32|64)` 设 `WINEARCH`；新 prefix 用 `wineboot` + `wineserver -w`；检测是否存在 `drive_c/windows/syswow64` 判定 win64。
- WoW64 模式：win64 下还区分 classic WoW64（32 位 wine + 独立 wine64）与 new WoW64（单一 wine 二进制）。
- 架构相关警告：`w_package_warn_win64()`、`w_package_broken_win64()`、`w_package_unsupported_on_win64()`、`w_package_broken_wow64()`。多数 `load_*` 在 win64 下会额外安装 x64 版。

## 7. 典型修复 Verb 举例

### vcrun2019（Visual C++ 2015–2019 运行库）
1. `w_override_dlls native,builtin` 一批 CRT DLL（msvcp140、vcruntime140、concrt140 等）。
2. 下载 `vc_redist.x86.exe`，cabextract 替换 msvcp140 再静默安装。
3. win64 下同样处理 x64 + `vcruntime140_1`。
4. `w_call ucrtbase2019`。冲突：vcrun2015/2017/2022 等。

### dotnet48（.NET Framework 4.8）
1. 检查 Wine bug、`w_package_warn_win64`。
2. `remove_mono` → `dotnet40` → `w_set_winver win7`。
3. 静默运行 `ndp48-x86-x64-allos-enu.exe`（`WINEDLLOVERRIDES=fusion=b`）。
4. `w_override_dlls native mscoree`。验证用 touch 标记文件。

### d3dx9（DirectX 9 运行库）
`helper_directx_Jun2010` → 从 DirectX June 2010 redist cabextract 出 `d3dx9_24`–`d3dx9_43.dll` → `w_override_dlls native d3dx9_*`。

### dxvk（用 Vulkan 实现 D3D8/9/10/11）
`helper_dxvk` 下载 tar.gz → 解压 mv x32/x64 DLL → `w_override_dlls native dxgi d3d8 d3d9 ...`。
> 注意：Proton 自带 DXVK/VKD3D，Steam 游戏通常**不需要**再装 winetricks dxvk。

### corefonts（微软核心字体）
依次 `w_call` 各字体 verb（cabextract + `w_register_font`），避免缺字体导致安装程序静默失败。

## 8. Winetricks 在 Proton 生态中的角色

Winetricks 负责"补齐 Windows 生态缺失组件"，但不解决"用哪个 prefix / 哪个 wine"的问题。这一层由 Protontricks 或 umu-protonfixes 完成：

```python
# protontricks 调用方式（cli/main.py）
command = [str(winetricks_path)] + winetricks_command   # 例：vcrun2019 d3dx9
```
Protontricks 的 `run_command()` 会预先设置 `WINEPREFIX`（游戏 compatdata prefix）、`WINE`/`WINESERVER`（Proton 自带）、`WINEDLLPATH`、`PATH`、Steam Runtime 容器等，再交给 winetricks 执行。

等价的手动操作：
```bash
export WINEPREFIX=~/.steam/steam/steamapps/compatdata/<APPID>/pfx
export WINE=<proton_dist>/bin/wine
winetricks -q vcrun2019 d3dx9
```

三层分工：
- **Proton**：提供 Wine 运行时与图形栈（DXVK/VKD3D）。
- **Winetricks**：补齐 Windows 生态缺失组件（VC++/.NET/字体/特定 DLL）。
- **Protontricks / umu-protonfixes**：解决 Steam prefix / Proton / Runtime 的发现与环境注入。

## 10. 设计要点总结

1. **单文件 monolith**：全部 verb 与基础设施在一个 shell 脚本里，无编译依赖，便于被 Proton 项目 vendoring。
2. **Verb = 声明式 metadata + 命令式 load**：易扩展（含外部 `.verb`），`w_call` 子 shell 隔离防止环境污染。
3. **两层修复**：文件层（DLL/字体/程序进 prefix）+ 配置层（DllOverrides/winecfg/注册表）。
4. **缓存与校验分离**：`~/.cache/winetricks` 全局共享，prefix 内 `winetricks.log` 记录 per-game 安装历史。
5. **与 Proton 分工明确**：Winetricks 只管"装什么"，"在哪装、用哪个 wine"交给 Protontricks / umu-protonfixes。
